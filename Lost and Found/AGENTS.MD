# 失物招领 APP — 产品需求文档

> 本文档用于《移动应用开发课程设计》，技术栈 前端为 Vue + UniApp ，后端为 Node.js + Express，文档定位为产品规划阶段的可执行参考。

---

## 一、目标用户

| 角色 | 描述 | 核心诉求 |
|------|------|---------|
| **失主** | 丢失物品、急于找回的学生 | 发布快、传播广、第一时间联系上拾主；联系方式隐私受保护 |
| **拾主** | 捡到物品的学生或教职工 | 归还意愿强但怕麻烦；发布门槛要低，不被过度打扰 |
| **管理员** | 学生会或后勤部门人员 | 审核异常信息、处理争议、管理敏感数据可见性；低频使用，重权限 |

---

## 二、核心场景

按使用优先级排列：

1. **发布失物** — 学生丢失物品后，打开 APP 填写名称、地点、时间、联系方式，选择配图，提交发布。
2. **发布招领** — 学生捡到物品后，拍照填写信息发布招领，系统自动检索失物库尝试匹配。
3. **主动搜索** — 用户在首页搜索关键词，按地点/类型/时间筛选，浏览结果后联系对方。
4. **匹配通知** — 系统检测到失物与招领信息高度吻合，自动推送给双方，促成联系归还。
5. **个人中心** — 用户查看自己的发布记录、信息状态（寻找中/已找到/已关闭）及匹配通知列表。

---

## 三、主业务流程

### 流程一：失物 → 匹配 → 找回

```
失主发布失物 → 信息写入失物库 → 系统检索招领库进行关键词匹配
→ 有匹配则推送给双方 → 失主联系拾主确认 → 任一方标记"已找到" → 结束
→ 无匹配则信息持续留存，新招领入库时再次尝试匹配
```

### 流程二：招领 → 匹配 → 归还

```
拾主发布招领（带图）→ 信息写入招领库 → 系统检索失物库进行匹配
→ 有匹配则推送 → 拾主与失主联系 → 归还完成标记
```

### 流程三：搜索 → 浏览 → 联系

```
用户输入关键词 → 筛选条件（地点/时间/类型）→ 浏览结果列表
→ 进入详情页查看联系方式 → 线下沟通
```

### 流程四：注册 → 登录 → 使用

```
首次打开 APP → 注册/登录（手机号或学号）→ 可选完善资料 → 进入首页
```

## 四、页面清单

共 5 个核心页面，按底部导航栏组织：

| 页面 | 核心内容 |
|------|---------|
| **首页** | 搜索框、失物/招领 Tab 切换、信息瀑布流列表、右下角悬浮"+"发布按钮 |
| **发布页** | 共用失物/招领表单：物品名称、类型选择、地点、时间、图片上传（最多 4 张）、联系方式、详细描述（选填）、提交按钮 |
| **详情页** | 图片轮播、物品信息、发布者信息、联系方式一键复制/拨打、"标记已找到"入口（仅发布者可见）、匹配推荐区域 |
| **搜索页** | 搜索框、筛选面板（时间/地点/类型/发布类型）、结果列表、空状态提示 |
| **个人中心** | 头像与基本信息、我的发布（失物+招领，按状态分 Tab）、匹配通知列表、设置入口 |

## 五、MVP 范围

### P0（必须做，否则产品不可用）

- 注册 / 登录
- 发布失物信息（名称、地点、时间、联系方式、图片）
- 发布招领信息（同上）
- 首页信息列表展示（失物 + 招领，按时间倒序）
- 关键词搜索
- 信息详情页
- 基本匹配逻辑（物品名称 + 地点关键词匹配，不要求语义理解）
- 匹配结果站内通知（双方可见）
- 标记"已找到"

### P1（建议做，体验加分）

- 图片上传
- 按地点和类型的分类筛选

> 以上 P0 功能可跑通"发布 → 浏览 → 匹配 → 联系 → 结束"的完整闭环。

## 六、明确排除清单（MVP 不做）

| 不做 | 原因 |
|------|------|
| 手机短信 / 推送通知 | 需第三方资质和付费，MVP 用站内通知即可 |
| 地图选点 / LBS 定位 | 需接入地图 SDK，校园场景文字描述更实用 |
| 智能语义匹配 / AI 识别 | 技术坑深，课程设计时间不足以踩坑 |
| 在线聊天（IM） | 等于再做一套聊天系统，成本远大于价值 |
| 信用评价 / 实名认证 | 需对接教务系统，复杂度高且非核心 |
| 悬赏 / 价值评估 | 与"找东西"核心业务无关 |
| 后台数据统计面板 | 锦上添花，先把用户核心流程跑通 |
| 第三方登录（微信/QQ） | MVP 用手机号注册即可，减少 SDK 接入 |

## 七、项目开发规则

> 以下规则适用于本项目全部代码编写阶段，前端与后端必须严格遵守。

| 序号 | 规则 | 说明 |
|------|------|------|
| 1 | **Vue 3 + Composition API** | 前端所有页面和组件必须使用 Vue 3 `<script setup>` 语法和 Composition API，禁止使用 Options API |
| 2 | **页面职责单一** | 每个页面仅负责一个核心业务，不得将多个业务逻辑混在同一页面文件中 |
| 3 | **禁止未使用代码** | 禁止保留未使用的变量、未使用的 import、未引用的文件或空文件，保持代码仓库干净 |
| 4 | **API 统一封装** | 所有 HTTP 请求必须通过统一的 `api/` 模块封装（如 `request.js` + 各模块 api 文件），页面中禁止直接调用 `uni.request` 等底层方法 |
| 5 | **后端分层架构** | 后端必须按 `controller（控制器）` → `service（服务层）` → `model（数据模型）` 三层结构组织代码，不得在 controller 中直接操作数据库 |
| 6 | **RESTful 接口风格** | 所有后端接口必须遵循 RESTful 规范：使用合适的 HTTP 方法（GET/POST/PUT/DELETE），URL 使用名词复数形式，资源间关系通过 URL 层级表达 |
| 7 | **统一返回格式** | 所有接口返回数据必须使用统一 JSON 结构：`{ code: 200, message: "success", data: {} }`，`code=200` 表示成功，其他值表示异常 |
| 8 | **禁止复杂 AI/NLP/图像识别** | MVP 阶段禁止引入任何 AI 语义匹配、NLP 分词、图像识别等复杂逻辑，匹配仅使用基础关键词比对 |
| 9 | **必须实现分页机制** | 所有列表类接口必须支持分页参数 `page` 和 `page_size`，返回数据包含 `total`、`page`、`page_size` 字段 |
| 10 | **代码必须带中文注释** | 所有自定义函数、复杂逻辑、数据结构定义必须添加中文注释说明用途，方便课程设计答辩讲解 |
